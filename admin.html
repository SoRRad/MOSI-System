🔄Submission Flow
1
Surgeon submits a case
They fill in the 4-step form on index.html and click Submit. No login, no token, no setup required from them.
2
Case saved locally (instant)
The case is written to the surgeon's browser localStorage immediately. This happens even if offline — data is never lost.
3
GitHub Actions workflow dispatched
The platform calls the GitHub API to trigger the submit_case.yml workflow using the admin token. This token is stored only in the admin browser and shared to the platform via localStorage — it never touches surgeon devices directly.
4
GitHub's servers run the workflow (~30s)
The runner reads data/cases.json, appends the new case, and commits it back. This uses GitHub's internal GITHUB_TOKEN — no credentials from surgeons needed.
5
Commit appears in your repository
Every case is permanently versioned with a timestamp. The Unit Aggregate view in the platform reads data/cases.json directly — all surgeons see the same live data.

📊R Analysis Code

Once you have cases in GitHub, load and analyse in R:

# Load directly from GitHub (public repo)
library(jsonlite); library(tidyverse)

url <- "https://raw.githubusercontent.com/owner/mosi-system/main/data/cases.json"
cases <- fromJSON(url)

# Agreement rates by stage
cases %>%
  group_by(stage) %>%
  summarise(
    n = n(),
    stage_agree = mean(stageAgreement == "Agree") * 100,
    proc_agree  = mean(procAgreement == "Agree") * 100,
    disagree_n  = sum(stageAgreement == "Disagree")
  )

# Or load from exported CSV
cases <- read_csv("MOSI_Cases_2026-03-17.csv")
